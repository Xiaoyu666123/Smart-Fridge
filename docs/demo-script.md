# 演示脚本

供录制答辩视频 / 现场 Demo / Spec 评审参考。**8 分钟版**完整覆盖所有亮点；**3 分钟版**用于答辩开场。

---

## 准备工作（演示前）

### 数据准备
1. 用 admin 账号登录，确保至少有 8-10 条 inventory 记录
2. 让端侧模拟器持续模式跑起来（演示视频背景里有"实时数据"）：
   ```cmd
   conda activate fastapi
   cd client/demo
   python luckfox_simulator.py loop --interval 30 --probability 0.4
   ```
3. 提前在 `/admin/categories` 给几个常见品类配上单价（比如苹果 ¥5、香蕉 ¥3、牛奶 ¥10）
4. 确保 admin / user 两个 token 都已登录（两个浏览器或两个无痕窗口）

### 录制工具建议
- OBS Studio（免费、能多源）
- Mac 自带 QuickTime
- 浏览器分辨率：**1920×1080** 或 **1440×900**，避开窄屏布局

## 8 分钟完整版

### 0:00-0:30 大屏开场

> **画面**：admin 顶栏点"大屏"胶囊 → 在新窗口打开 → F11 全屏

> **旁白**：本系统是端云协同的智能冰箱食材管理方案。这是实时监控大屏，左上角是渐变标题，右上角是大字时钟。下方五个 KPI 是总库存、即将到期、已过期、在线设备、浪费率，**所有数字实时变化**。

> **关键操作**：让 simulator 跑个 1-2 个 ITEM_IN 事件，演示**"实时事件流"**右下角滑入动画 + 趋势线翘起。

---

### 0:30-1:30 端云协同与标签缓冲

> **画面**：返回 admin → `/admin/inventory` → 顶栏快速识别 FAB（右下角圆按钮）

> **旁白**：项目最大的设计点之一是端云分工。LuckFox Pico Pro Max 算力有限，本地能跑 YOLO 检测，但读不清商品标签。我们设计了"标签缓冲"机制：用户先把标签朝向摄像头，端侧把标签照片传给后端，云端 OCR 解析品牌、保质期、生产日期，写入临时缓冲表（5 分钟 TTL）。当用户随后把物品放进冰箱，端侧 YOLO 触发 ITEM_IN，后端按时间窗自动配对最近一条未消费的标签，用真实保质期覆盖 LLM 的估算值。

> **关键操作**：进 `/admin/pending-labels` 演示缓冲条目；展开任意一条 inventory 看带 📦 标签徽标。

---

### 1:30-3:00 多 Agent 工具链 + AI 可解释

> **画面**：进 `/admin/workflow` → 选最新一条 ITEM_IN trace

> **旁白**：每条业务事件都会触发一条 Agent 工具链。这条 ITEM_IN 走过 6 个工具步骤：向量去重、视觉辅助识别决策、云端识别、保鲜期推算、标签关联、写入数据库。每个节点能拖拽，点击展开看 JSON 输入输出。耗时、成功率全部留痕。

> **关键操作**：点右上角"AI 解释决策"按钮 → 等 3-5 秒 → 看 LLM 把 6 步翻成自然语言段落。

> **旁白**：这就是"可解释 AI"。我们把传统的黑盒决策变成一段中文叙述，让用户和评审清楚"AI 是怎么做的"。

---

### 3:00-4:00 跨 tab 实时同步

> **画面**：开两个浏览器窗口（admin 后台 + admin 库存页）

> **旁白**：库存数据通过 WebSocket 全局广播。一个 tab 的操作，所有 tab 立刻同步。

> **关键操作**：
> 1. 在 tab A 添加一个食材 → tab B 看到新行**滑入 + 薄荷绿高亮**
> 2. 在 tab A 删除一个食材 → tab B 看到行**淡出消失**
> 3. 切到大屏 tab → 看右下角实时事件流**滚动**

---

### 4:00-5:00 浪费分析 + 智能购物清单

> **画面**：`/admin/waste`

> **旁白**：传统浪费率指标停留在数字。我们把数字升级为可执行的购物清单：每个品类配单价后，浪费分析自动估算金额。底部的 365 天日历热图把全年浪费一眼看清。

> **关键操作**：
> 1. 看顶部三张金额卡（浪费 ¥XX / 已消耗 ¥XX / 单价覆盖率）
> 2. 滚到补货建议 → 点"复制清单" → 切到记事本粘贴 → 看到完整购物清单（带预估价）
> 3. 滚到日历热图 → 鼠标悬浮看每天浪费金额

---

### 5:00-6:00 食材生命周期 Sankey

> **画面**：`/admin/lifecycle`

> **旁白**：这是项目的可视化亮点——食材生命周期 Sankey 流图。横向三列分别是来源、品类、终态。线的粗细代表食材数量。

> **关键操作**：
> 1. 鼠标悬浮"端侧识别"节点 → 高亮全部下游
> 2. 切换 30 天 / 90 天看不同窗口的差异
> 3. 看右下角"已浪费"节点的入流，分析浪费集中在什么品类

---

### 6:00-7:00 AI 营养教练 + 食谱推荐

> **画面**：切到 user 浏览器 → `/user/nutrition`

> **旁白**：用户端有一个 AI 营养教练。它把当前库存按蔬菜/肉/水果九大类做分布，配合"蔬果占比 / 蛋白质占比 / 零食占比"三维计算 0-100 分健康评分。

> **关键操作**：
> 1. 点"让 AI 给我建议"按钮 → 等 3-5 秒
> 2. 看到结构化建议：本周饮食安排 + 行动建议 + 避免食物
> 3. 切到 `/user/chat` → 点"清临期"快捷卡 → 看 LLM 流式输出 + 按 `===RECIPE===` 块解析的食谱卡片
> 4. 收藏一个食谱 → 切到 `/user/recipes` → 演示评分、笔记、打卡扣库存

---

### 7:00-8:00 可信 + 可控 + 端到端

> **画面**：`/admin/agent` 视觉辅助配置 → `/admin/audit` 审计日志

> **旁白**：所有 AI 决策都是可控的。视觉辅助识别按"区间触发"——只有端侧置信度落入这个区间才调云端，节省 token；超出则信任端侧。区间由管理员双滑块调整，每次调整自动记录审计。

> **关键操作**：
> 1. 拖动滑块演示区间调整
> 2. 切到 `/admin/audit` → 看刚才的操作被记录在审计日志时间线上
> 3. 顶部 KPI 看"今日操作"数字 +1

> **结尾旁白**：从端侧 YOLO，到云端识别，到 LLM 推理，到前端实时同步，再到管理员审计——本项目构建了一个完整闭环的智能冰箱平台，**用 30+ 个功能模块演示了多 Agent + 可解释 AI + 实时数据驱动**的端云协同范式。

## 录制小贴士

1. **不要展示登录过程**：直接打开已登录的页面，避免 token 输入耽误时间
2. **鼠标轨迹放慢**：录制时鼠标移动慢一些，观看者才看得清你点的什么
3. **关闭浏览器通知**：录制前关掉所有系统/浏览器通知避免突然弹出
4. **关掉无关 tab**：tab 栏只留演示用的几个
5. **暗色 / 浅色按需选**：暗色模式录大屏更酷，浅色更适合表格演示
6. **配字幕**：录完用剪映加字幕，旁白文字浮在画面下方
7. **音乐**：背景配一段轻快的电子乐（剪映免版权库够用），开头 0.5s + 结尾 1s 加 fade

## 常见 Q&A 准备

**Q：实时同步用 WebSocket 而不是轮询，理由？**
A：库存事件是稀疏但需要立刻可见的。轮询要么延迟高（10s+），要么资源浪费（1s 一次）。WS 长连接 + broadcast_all 是天然合适的方案。

**Q：双层去重，向量层为什么不在数据库做主键约束？**
A：向量相似度本质是模糊匹配，没有"严格相等"语义。主键只对字节哈希有效，向量层是逻辑层"判定"，方便业务调阈值。

**Q：为什么要做 AI 解释？**
A：可信 AI 是当前 AI 落地的核心问题。我们的 trace 数据本身就是结构化决策过程，让 LLM 把它翻译成自然语言能直接体现"可解释机器学习"的实践价值。

**Q：标签缓冲为什么用 5 分钟 TTL？**
A：用户从扫标签到放进冰箱通常 30 秒内完成。5 分钟是宽松上限，避免长时间无关物品误配对。TTL 后未消费的会被懒清理。

**Q：项目部署在云上吗？**
A：当前是单机演示版，但已经留好部署接口（FastAPI + Postgres + 静态文件，可以直接 docker-compose），多 worker 时 WS 跨进程需要 Redis pub/sub。
